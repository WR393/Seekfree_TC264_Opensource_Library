# 错误处理规范

> 这个仓库没有异常系统，也没有 Web 风格的错误返回链路。当前固件的错误处理原则是：先限幅、再清状态、必要时重试、绝不破坏实时性。

---

## 当前处理模式

代码里已经形成了四种主要模式：

- 输入或输出限幅
- 协议帧错误后立即清空解析状态
- 外设初始化失败时循环重试
- 中断里只做短逻辑并尽快回到稳定状态

---

## 现有硬约束

| 场景 | 当前代码事实 | 规范要求 |
|------|--------------|----------|
| 电机占空比 | `motor_set_duty()` 将输入夹到 `[-5000, 5000]` | 应用层先限幅，再下发协议层 |
| 电机速度 | `motor_set_speed()` 将输入夹到 `[-3000, 3000]` | 速度目标必须先做安全裁剪 |
| PID 输出 | `pid_out_limit()` 约束 `PID_T.out`，`pid_app_limit_integral()` 约束积分 | 控制算法必须显式定义输出上限和积分上限 |
| FOC 串口收包 | 7 字节帧，帧头或校验错就清 `receive_data_count` 和缓存 | 协议错误时立即丢弃当前帧，不保留半包状态 |
| IMU 初始化 | `imu_all_init()` 失败后 `system_delay_ms(100)` 再重试 | 对可能迟到的设备，用重试而不是假设一次成功 |

## ISR 安全规则

- 中断入口先完成必要的开中断与标志位清除，再做回调分发
- 中断里不写阻塞循环，不做复杂打印，不做大段业务判断
- 高频控制逻辑可以由固定节拍 ISR 触发，但具体算法应放在独立模块函数里
- 非关键工作下沉到前台调度或测试函数

当前节拍契约：

- 1ms：维护 `uwtick`
- 2ms：`imu_proc()` + `balance_gyro_loop()`
- 10ms：`balance_angle_loop()`
- 20ms：`balance_steering_loop(0.0f)`

如果改动这些频率，必须同步审视：

- PID 参数是否还成立
- `scheduler_run()` 的任务是否与中断写同一状态
- 调试输出是否会拉长中断时长

## 场景：协议解析错误恢复

### 1. Scope / Trigger

- 触发条件：UART 收包过程中收到坏帧、半帧、错帧头或错校验

### 2. Signatures

当前收包入口：

```c
void uart_control_callback(void);
```

### 3. Contracts

- 帧长固定为 7 字节
- 帧头固定为 `0xA5`
- 当缓存达到 7 字节时才做完整校验
- 校验失败、帧头错误、功能号不匹配时，清计数并清缓冲

### 4. Validation & Error Matrix

| 输入情况 | 必须行为 |
|----------|----------|
| 第一个字节不是 `0xA5` | 不把该包当成有效帧开头 |
| 帧头正确但校验错 | 清空缓冲并等待下一帧 |
| 功能号不是速度回传 `0x02` | 不更新速度结果 |
| 正常 7 字节速度帧 | 更新左右轮速度并清空缓冲 |

### 5. Good / Base / Bad Cases

- Good：任何坏帧都只影响当前帧，下一帧可正常恢复
- Base：功能号暂时只有少数几种，但错误功能号不会污染现有状态
- Bad：坏帧残留在缓存里，导致后续帧持续错位

### 6. Tests Required

- 发送正确速度帧，确认左右速度更新
- 故意破坏最后一字节校验，确认状态不会更新
- 连续注入错帧头后再发一帧正确包，确认能够恢复

### 7. Wrong vs Correct

#### Wrong

- 在坏帧后保留已有 `receive_data_count`
- 让上层应用自己猜测半包是否有效

#### Correct

- 协议层自己清状态，向上层暴露干净的解析结果

## 常见错误

- 未限幅就把控制值发到驱动层
- 在 ISR 里加入 `printf()` 或其他慢路径
- 中断标志还没清就开始写业务逻辑
- 前台和中断同时改同一份控制状态，但没有时序约束

## AI Drift Guardrails

- 协议层已经发现坏帧时，不要把半包恢复责任继续往 app 层外推；清计数、清缓存、恢复同步仍由协议 owner 自己完成。
- 不要为了“更方便调试”就在错误恢复路径里插入高频 `printf()`；先恢复稳定状态，再决定是否在前台补观测。
