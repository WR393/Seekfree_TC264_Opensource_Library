# 质量规范

> 这个仓库的质量目标不是“抽象优雅”，而是“时序稳定、边界清楚、容易在板上验证”。

---

## 必须遵守的模式

- 模块头文件优先统一包含 `zf_common_headfile.h`
- 对外 API 放 `.h`，实现放配套 `.c`
- 函数名带模块前缀，如 `balance_init`、`motor_set_speed`、`pid_set_target`
- 全局状态必须有明确物理含义或协议含义
- 有风险的输入、输出、积分项、收包状态都要显式限幅或清理
- 分层必须清楚：入口层、应用层、驱动/协议层、平台层

真实样例：

- `code/app/motor_app.h` 和 `code/app/motor_app.c` 暴露小而明确的电机 API
- `code/driver/pid/pid_driver.h` / `.c` 保持算法接口和实现一一对应
- `code/app/balance_app.c` 复用 `PID_T`，而不是复制一份 PID 公式
- `code/driver/motor/small_driver_uart_control.c` 吞掉字节流细节，让 `motor_app.c` 保持应用语义

---

## 分层边界要求

- `user/` 只放启动和中断入口，不放大段业务流程
- `code/app/` 负责“要做什么”，不负责“字节怎么发”“寄存器怎么配”
- `code/driver/` 负责“怎么和设备/算法交互”，不负责上层策略
- `libraries/` 默认只复用，不优先当作业务代码编辑区

---

## 头文件与命名要求

- 新模块默认一对 `.h + .c`
- 头文件保护宏跟随路径风格
- 不随意引入新的公共入口头文件，优先对齐现有 `zf_common_headfile.h`
- 注释优先解释物理意义、单位、调用周期，不写无信息量注释

---

## 变更验证矩阵

| 改动类型 | 至少要验证什么 |
|----------|----------------|
| 启动顺序改动 | 外设初始化顺序、依赖关系、主循环正常进入 |
| 中断改动 | 标志位处理、回调路径、频率、是否引入阻塞 |
| PID / 控制改动 | 限幅、方向、输出范围、实际节拍 |
| 协议改动 | 帧头、长度、校验、异常恢复 |
| 全局状态改动 | owner 是否唯一、是否与 ISR/前台冲突 |
| Flash / 配置改动 | 页面布局、默认值、恢复路径 |

---

## 板级测试要求

当前仓库没有可见的单元测试框架，因此变更后的最低验证是：

- ADS 工程能正常编译
- 板级启动正常
- 相关中断路径能按预期触发
- 传感器符号、控制方向、执行器方向正确
- 调试输出没有把实时路径拖慢

如果改的是控制或协议逻辑，提交前至少记录清楚：

- 初始化是否成功
- 回调是否命中
- 限幅是否生效
- 错帧或异常输入时是否恢复
- 关键物理量的正负号是否正确

---

## 评审清单

- 代码是否放在正确层级
- 是否复用了现有模式，而不是新造一套结构
- 是否显式处理了限幅、坏输入和恢复路径
- 中断与前台是否有清晰时序边界
- 名称、注释、变量是否表达物理意义
- 如果新增持久化，是否已经把布局和恢复规则写进规范

---

## 明确禁止

- 在应用层重复实现驱动协议
- 在 `user/isr.c` 或 `user/cpu0_main.c` 里塞入大量业务逻辑
- 为了省事直接改厂商库实现应用需求
- 在高频路径保留常驻调试打印
- 复制现有模块然后各改一点，造成多份近似逻辑并存

---

## 常见错误

- 调度逻辑、中断逻辑、启动逻辑混在同一个文件里
- 安全限幅只写在某个调用点，导致别的调用点漏掉
- 上层直接依赖底层缓存结构，破坏模块边界
- 改了周期或符号方向，但没重新核对控制参数

## AI Drift Guardrails

- 当前平衡控制语义默认以 `pitch` 为主；除非任务明确要求更换控制模型，否则不要把新增控制逻辑改写成 `roll` 主导。
- 前台调度与高频控制环必须保持边界：`scheduler_run()` 负责可延后任务，PIT 节拍负责实时控制。
- 如果改到姿态/控制链路，评审时必须确认仍以 `pitch` 为主控制量，而不是被遥测里的 `roll` 语义带偏。
- 如果改到协议或观测链路，评审时必须确认收包状态 owner 仍在 driver 层、日志仍不进入高频 ISR。
- 新增模块时绕过 `zf_common_headfile.h`，重新发明一套公共入口头文件，视为违规。
- 把协议收包缓存、帧计数、校验恢复逻辑泄漏到 `code/app/` 或 `user/isr.c`，视为违规。
- 因未来可能接 PC 工具，就提前把当前固件规范改成混合 frontend/backend 的目录思路，视为违规。
- 把 `roll` 的遥测存在误读成“控制也应该改用 roll”，属于典型误判。
